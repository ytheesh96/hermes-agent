Hermes Loop Design Note

Portable Loop Context and Audit Trail

A future-facing model for making Loops durable, recoverable work objects that can be attached to clean sessions without losing the audit trail or allowing two foreground agents to act on the same Loop at once.

Created: 2026-06-18 Scope: Hermes Desktop / Loop / Kanban Status: design reference

Executive Summary

Treat the Loop as the durable work object and the session as a disposable conversation surface. If a session becomes corrupted, the user should be able to continue the Loop from a clean session by hydrating the Loop overview drawer from durable Kanban/Loop state.

The root task description should remain the durable context contract: goal, user decisions, constraints, current state, open questions, and acceptance criteria. The audit trail should remain structured under the Loop: child tasks, worker runs, events, handoffs, artifacts, and foreground decisions.

Core Model

Loop

The durable unit of work. Owns the goal, root task description, task graph, handoffs, evidence, and audit trail.

Session

A temporary view or conversation over the Loop. Useful, but not the source of truth.

Foreground Agent

The active controller that can make foreground decisions for the Loop. Only one foreground session should control a Loop at a time.

Primary Decision

Decision Recommendation Reason
Recovery surface Hydrate the Loop overview drawer. The drawer already contains the relevant work state without copying the old transcript into the new chat.
Main preserved context Use the root task description as the durable contract. User decisions already live there, so the root description can be trusted as the canonical working brief.
Concurrency Allow many viewers, but only one active foreground controller. Prevents two sessions from independently acting on the same Loop and creating split-brain state.
Old sessions Link them as searchable evidence, not primary context. Old foreground sessions are useful for details and provenance, but should not be dumped into every future context window.

What the Root Description Should Preserve

Durable Contract Sections

  • Goal: what this Loop is trying to accomplish.
  • User decisions: choices made by Vaitheesh.
  • Constraints: safety, scope, permissions, and exclusions.
  • Current state: what is true now.
  • Open questions: unresolved choices or blockers.
  • Acceptance criteria: what done means.
  • Evidence: task ids, run ids, artifacts, handoffs.

Provenance to Add

  • User decision source: foreground session id or message id.
  • Worker evidence source: task id, run id, worker session id.
  • Foreground decision source: reviewer session id and review run id.
  • Artifact source: file path plus producing task/run.

Audit Trail Advantages

Why Loops Beat Transcripts

  • Structured work graph instead of chronological chat noise.
  • Recoverable even if one session gets corrupted.
  • Can trace work to task, run, worker profile, worker session, and artifact.
  • Foreground approvals and escalations become durable decisions.
  • Drawer hydration can stay compact and inspectable.

Traceability Chain

The audit identity should use the concrete run/session, not only the assignee profile.

Concurrency Rule

Invariant

A Loop may be attached to many sessions for viewing, but exactly one foreground session may hold the active controller lease.

Allowed

  • Multiple sessions attach the same Loop in read-only mode.
  • The active session controls foreground decisions.
  • A user transfers control with "Continue elsewhere".
  • A user force-takes control when the old session is dead.

Prevented

  • Two foreground agents independently resolving the same blocker.
  • Two sessions racing to approve or escalate the same handoff.
  • Silent ownership drift after session recovery.
  • Old session continuing to act after control moved elsewhere.

Attach and Continue Flow

Step 1

Select Loop

Choose "Attach Loop" or "Continue in new session" from the drawer.

Step 2

Hydrate Drawer

Load root, children, handoffs, artifacts, evidence, and open questions.

Step 3

Check Lease

If another session owns control, attach as view-only or request takeover.

Step 4

Transfer Control

Continue elsewhere moves the active foreground lease to the new session.

Step 5

Record Event

Audit the attach/transfer so future agents know what happened.

UI Recommendations

Drawer Actions

  • Continue in new session: creates a clean session and transfers control.
  • Attach to current session: hydrates the drawer in the active chat.
  • View linked sessions: shows origin and recovery sessions.
  • Take over: explicit user action when a stale owner blocks progress.

Session Surface

  • Show a compact attached Loop chip in the composer/header.
  • Show whether this session is active controller or view-only.
  • Warn before acting when another session owns control.
  • Do not paste the full root description into chat by default.

Pros and Cons

Pros

  • Clean recovery from corrupted sessions.
  • Durable audit trail for worker work and foreground decisions.
  • Less context bloat than transcript copy-forward.
  • Better accountability through task/run/session lineage.
  • Natural support for "continue elsewhere".
  • Root task description remains the canonical work brief.

Cons and Mitigations

  • Stale overview: require update timestamps and provenance.
  • Concurrency: enforce one active foreground lease.
  • UI ambiguity: distinguish attached, view-only, and controlling states.
  • Context bleed: link old sessions as searchable evidence only.
  • Bad summaries: keep audit events and artifacts traceable.

Implementation Phases

  1. Phase 1: Attach and hydrate. Add a session-to-Loop link and hydrate the Loop overview drawer from durable Loop/Kanban state. Add a compact chat marker.
  2. Phase 2: Active foreground lease. Enforce one active controller per Loop with heartbeat, expiry, transfer, and force-takeover.
  3. Phase 3: Linked session search. Store linked foreground sessions and allow future foreground agents to search them when the root description is insufficient.
  4. Phase 4: Provenance polish. Add UI affordances showing where each decision, artifact, and handoff came from.

Acceptance Criteria

  • A user can continue a Loop in a new clean session without copying the old transcript.
  • The new session shows the hydrated Loop overview drawer with root, children, handoffs, artifacts, decisions, and open questions.
  • The root task description remains the canonical user-decision contract.
  • Only one foreground session can actively control the Loop at a time.
  • View-only attached sessions cannot approve, escalate, or mutate foreground handoffs.
  • Every control transfer or takeover is auditable.
  • Actual worker work remains traceable to task_id, run_id, worker_profile, worker_session_id, and artifacts.

Recommended Next Step

Build the smallest useful version first: "Continue in new session" from the Loop drawer. It should create or focus a clean session, attach the Loop, hydrate the drawer, and transfer the active foreground controller lease. Old sessions should become view-only for that Loop.
